Management summary

My proposal is simple: when the risk belongs to shared XUI behaviour, XUI should prove that risk centrally instead of asking every onboarded service to rediscover the same problem through broad SRT. This POC shows that we can take real EXUI configuration, real service-definition evidence, historic failure classes, and accessibility concerns, then turn them into a small set of executable assurance slices with inspectable Odhín evidence.

I am not claiming this replaces all service testing. Service-owned business journeys, bespoke content, documents, and unusual definitions still need service ownership. What I am saying is that we now have a working route to remove repeat testing where services are only proving the same EXUI rule from different angles.

The important bit is the proof: the harness has a green run, accessibility baseline gates, screenshots, API and UI evidence, and mutation runs that deliberately break shared EXUI and CCD-definition assumptions so we can show the gate going red. That makes this much more than a theoretical proposal.

What changes: central proof for shared XUI risk, not repeated broad SRT for every service.
How it works: source truth becomes representative slices, then Playwright proves the contract.
What is new: accessibility is now a first-class baseline lane, not a manual afterthought.
What comes next: service teams review the slice catalogue and call out shapes we have missed.
Proposal in practice

I am not selling a blanket replacement for service testing. I am selling a more sensible split of responsibility. If XUI owns the behaviour, XUI should prove it centrally. If a service owns the journey, callback, content, or data rule, that still belongs with the service.

The value is that we stop asking thirteen teams to prove the same shared XUI rule in thirteen slightly different ways. We prove that rule once, with source evidence, tests, screenshots, accessibility evidence and a report people can actually inspect.

1. The basic idea

Inputs

Real EXUI and service truth

  • EXUI config and API route behaviour from rpx-xui-webapp
  • CCD definition evidence from service repos, starting with PRL
  • Historic SRT and incident failure classes
Model

Central assurance representation

  • Source snapshot JSON for traceability
  • Executable TypeScript model imported by tests
  • Representative service slices, not every journey
Evidence

Playwright proof

  • API contract checks
  • Thin UI checks where behaviour is visible
  • Mutation proof that deliberately makes the gate fail
The point: we are not saying one PRL test proves every PRL business journey. We are proving shared EXUI interpretation of configuration shapes so services do not all need to retest the same EXUI behaviour independently.
The important shift

The harness is organised around reusable XUI behaviours, not around service names. A service may contribute one or many slices, but the test only earns its place if it proves a distinct XUI interpretation path.

Kept

Why these survive

  • They change EXUI-visible family membership, tab visibility, filters, hearings actions, or entitlement behaviour.
  • They map to known SRT risk areas: WA availability, hearings, role/access, CYA, hidden data, and search/workbasket metadata.
  • They are backed by concrete PRL definition evidence refs in EXUI_PRL_NORMALIZED_SLICES.
Grouped away

Why most files are not one test each

  • Many rows are service business variations that hit the same EXUI rendering path.
  • Repeated field/event definitions do not create new EXUI behaviour unless show conditions, hidden fields, roles, or tabs differ.
  • Service callback logic and business validation remain service-owned unless the risk is specifically EXUI interpretation.
Not thrown away

What becomes explicit gap or next slice

  • Event-page/CYA data-loss risks need a promoted CCD-backed browser or toolkit replay slice before full journey replacement claims.
  • Every PRL journey is not claimed as covered.
  • If a definition pattern introduces a new EXUI interpretation shape, it becomes a new slice.
The PRL argument: the POC compresses 1,475 PRL CCD definition files into a small number of EXUI-relevant behaviour lanes. That is the mechanism for SRT reduction: service variation is grouped only when EXUI would process it through the same shared path. This does not mean one PRL slice proves all of PRL.
What is not being thrown away

The files we do not turn into individual tests are not ignored. They are grouped only when they hit the same XUI behaviour. If a definition introduces a different show/hide rule, hidden-data rule, complex-type shape, role condition, tab rule, search metadata shape or accessibility-relevant DOM shape, that becomes a new slice or an explicit gap.

6. Hard questions: what is actually covered, what is a lane, and who signs it off?

This is the clarification that should stop the proposal being oversold. The current POC proves the mechanism. The next delivery phase is the work of building enough source-backed slices, with service-owner review, to make central evidence trustworthy lane by lane.

Current POC

Not exhaustive yet

  • PRL is the first real evidence source, not a claim that all PRL journeys are covered.
  • One slice per service is not enough for complex services.
  • The POC proves green evidence, red mutation evidence, and the source-to-slice model.
  • It does not remove SRT by itself.
Lane

A risk area, not a service

  • A lane is a shared EXUI-owned behaviour area.
  • Examples: work-allocation, hearings, case-list-search, manage-case, cya-rendering, auth-session.
  • A green lane does not automatically cover another lane.
  • A service usually maps to several lanes.
Service review

Acceptance, not blind trust

  • EXUI/harness team proposes a slice with source files and assertions.
  • Service team confirms whether the shape is real and representative.
  • Test lead accepts, rejects, or asks for another slice.
  • Only accepted lanes can reduce repeated SRT.
Question
Honest answer
Decision rule
How many slices for PRL?
Rough planning band: 10-14 slices before PRL EXUI-owned risk starts to look defensible. The current POC has only the first PRL slice family and selected lanes.
Add slices for distinct EXUI interpretation shapes: CYA/show-hide, hidden data, search/workbasket, WA, hearings, roles, event history, auth and protected endpoints.
How many for Civil or another large CCD service?
Likely 12-18 slices because the definition surface and case-type variation are broad.
Do not copy every Civil journey. Scan for shapes that EXUI interprets differently from PRL or existing slices.
How many for IA, SSCS, Employment or similar?
Likely 5-10 slices each, depending on hearings, case types, role clusters and definition complexity.
Reuse existing slices where the EXUI path is equivalent. Add new slices only where the shape differs.
What about Crime?
Unknown until source discovery. Crime is not currently part of the central EXUI family model inspected for this POC.
First step is discovery/classification: source repo, EXUI family membership, case types, roles, IDAM/CCD patterns, then slice decisions.
What about CCD itself?
Create a cross-service CCD interpretation pack, probably 12-15 slices, covering metadata and event/page/field interpretation patterns.
Cover CCD shapes once centrally when they hit shared EXUI logic: CaseType, SearchInputFields, WorkBasketResultFields, CaseEventToFields, ComplexTypes, authorisations, display contexts and show conditions.
What about IDAM/session?
Create an auth/session pack, probably 7-9 slices, covering login callback, session cookie, logout, role hydration, protected endpoints and internal/external persona differences.
Do not mix this with service journey proof. Treat IDAM as its own cross-cutting EXUI risk lane.
Most important wording: the POC proves the assurance harness mechanism, not exhaustive PRL assurance. The implementation work is to turn this into a source-scanned slice catalogue where every configured service family, CCD shape, role cluster and EXUI seam is covered, grouped, canary, or explicitly out of scope.
How service teams stay involved

The harness team should not decide in isolation that a slice represents a service. The sensible model is: XUI proposes the slice and shows the source evidence, the service confirms whether it is a real representative shape, and the test lead decides whether that evidence is good enough to reduce repeated SRT for that lane.

7. How a service becomes a slice

1

Find EXUI membership

Which configured sets mention this family: global search, WA, staff-supported, hearings, jurisdiction/case type maps?

2

Prove it is real

Find service repo evidence: CCD jurisdiction, case types, service codes, roles, event/page/field patterns.

3

Classify the shape

Candidate gate, grouped, canary, source-gap, or source-unavailable. New configured families must not silently disappear.

4

Add scenario

Each scenario has lane, service family, role cluster, assertion, execution mode, and priority.

5

Pick proof level

API first. Add UI only for visible behaviour. Promote to browser plus CCD only when that is the risk.

PRL today: PRL is the first normalized service slice. It uses real PRL CCD definition evidence, but it deliberately does not copy all PRL definition files or claim every PRL journey is covered.
The rule for adding more services

A new service does not automatically mean a pile of new tests. First we ask whether it uses a shape the harness already covers. If it does, it can be grouped with evidence. If it does not, we add a new slice with the source files, the assertion, and the proof level that matches the risk.

9. Developer instructions: run the harness manually

This section is for developers and test leads who want to inspect the proof themselves. The harness can be reviewed and run directly from the repos using normal shell commands.

What a reviewer should expect

The green run should show API, UI, integration and accessibility lanes in the Odhín report. The red runs should fail on purpose and name the broken contract clearly. If the red run does not fail, the harness is not proving the risk. If it fails vaguely, the evidence is not good enough yet.

Required checkouts

Use a sibling repo layout

  • rpx-xui-e2e-tests: the harness, tests, reports, and mutation wrappers.
  • rpx-xui-webapp: EXUI config source and local Node/API shell.
  • prl-ccd-definitions: PRL source evidence for the first normalized CCD slices.
  • ccd-docker: optional for future real CCD-backed browser journeys; not required for the current local demo proof.
Local runtime

What the demo starts

  • EXUI backend mock on 8080.
  • EXUI Node API on 3001.
  • EXUI static shell on 3455.
  • Synthetic seams on 4096 and 8091 for local-only dependencies.

Before you start

Tools

Use Node.js compatible with the repos, Corepack/Yarn, Git, and Chromium via Playwright. On a standard developer Mac this should be enough for the current proof.

Credentials

The local proof uses mocked or synthetic seams for the current POC, so no live case data or service credentials are needed for the demo run.

Ports

The demo uses 8080, 3001, 3455, 4096, and 8091. Stop conflicting local processes first, unless they are the same compatible harness services.

CCD Docker

The current shareable proof does not require the full CCD Docker stack. Use ccd-docker only when extending the POC into a real CCD-backed event-page or CYA browser journey.

1. Prepare the workspace

The demo runner assumes rpx-xui-e2e-tests, rpx-xui-webapp, and prl-ccd-definitions are checked out next to each other under the same parent folder.

mkdir -p xui-assurance-demo
cd xui-assurance-demo
export XUI_HARNESS_WORKSPACE="$(pwd)"

git clone git@github.com:hmcts/rpx-xui-e2e-tests.git
git clone git@github.com:hmcts/rpx-xui-webapp.git
git clone git@github.com:hmcts/prl-ccd-definitions.git

cd "$XUI_HARNESS_WORKSPACE/rpx-xui-e2e-tests"
git checkout test/harness-shareable-clean-history

2. Install dependencies

Use the repo-managed Yarn versions. The harness needs Chromium for the UI proof; EXUI needs a local Angular build before the shell can be served.

cd "$XUI_HARNESS_WORKSPACE/rpx-xui-e2e-tests"
corepack enable
yarn install --immutable
yarn test:setup:playwright-install-chromium

cd "$XUI_HARNESS_WORKSPACE/rpx-xui-webapp"
corepack enable
yarn install --immutable
yarn build:dev

3. Run the source/config drift gate

This checks that the harness model still matches current EXUI config and that the PRL source anchors used by the normalized slices still exist.

cd "$XUI_HARNESS_WORKSPACE/rpx-xui-e2e-tests"
yarn harness:manifest

4. Start the local demo runtime

Keep this terminal open. The command starts or reuses the local services required by the current POC. If a port is already owned by a compatible service, the runner will reuse it.

cd xui-assurance-demo
export XUI_HARNESS_WORKSPACE="$(pwd)"
cd "$XUI_HARNESS_WORKSPACE/rpx-xui-e2e-tests"
TEST_ENV=local TEST_URL=http://localhost:3455 yarn harness:demo:start

5. Run the green proof and open the report

Use a second terminal for the proof run. This produces the Odhín report that test leads and developers should review.

cd xui-assurance-demo
export XUI_HARNESS_WORKSPACE="$(pwd)"
cd "$XUI_HARNESS_WORKSPACE/rpx-xui-e2e-tests"
TEST_ENV=local TEST_URL=http://localhost:3455 yarn harness:local:odhin
open test-results/harness-poc-odhin-report/harness-poc-odhin.html

6. Run the controlled red proofs

These commands are useful when explaining why the harness is not just a set of happy-path checks. Each wrapper runs a good control first, then injects one targeted failure and expects the harness to catch it.

cd xui-assurance-demo
export XUI_HARNESS_WORKSPACE="$(pwd)"
cd "$XUI_HARNESS_WORKSPACE/rpx-xui-e2e-tests"
TEST_ENV=local TEST_URL=http://localhost:3455 yarn harness:mutation:wa
open functional-output/tests/harness/mutation-proof/odhin-report/harness-mutation-proof.html

TEST_ENV=local TEST_URL=http://localhost:3455 yarn harness:mutation:ccd
open functional-output/tests/harness/ccd-mutation-proof/odhin-report/harness-ccd-mutation-proof.html

What to look at in the report

Green central proof

Look for config contract checks, global search families, WA-supported families, staff-supported families, historic replay packs, Manage Tasks UI, and Hearings UI.

UI screenshots

The UI proofs attach screenshots for the Manage Tasks service-family filter, supported PRL hearings actions, and unsupported-family hidden Hearings tab.

Accessibility reports

The representative UI slices now include explicit accessibility baseline: tests. The reports show current known axe debt and fail if a future EXUI change introduces a new rule violation or increases the recorded baseline.

WA mutation proof

The red proof should name the missing shared service family, for example PRIVATELAW, rather than failing with a vague UI symptom.

CCD mutation proof

The red proof should name the missing CCD-origin search metadata field, for example [CASE_REFERENCE].

Common setup failures

Use this as the first triage list when the local proof does not start cleanly. These are setup problems, not product failures.

Symptom EXUI Angular build is missing
Likely cause rpx-xui-webapp has not been built locally.
Fix Run yarn build:dev in rpx-xui-webapp.
Symptom EXUI shell is not reachable
Likely cause The local demo runtime is not running, or port 3455 is blocked.
Fix Run yarn harness:demo:start and keep that terminal open.
Symptom Manifest fails on EXUI config
Likely cause The harness snapshot no longer matches rpx-xui-webapp config.
Fix Review the EXUI config change and update the harness source snapshot/model if the change is intentional.
Symptom Manifest fails on PRL evidence
Likely cause prl-ccd-definitions is missing, not next to the harness repo, or the PRL definition shape moved.
Fix Check out prl-ccd-definitions next to the harness repo, or update the source anchors if PRL changed intentionally.

10. Accessibility in the assurance harness

Accessibility should be treated as a first-class harness lane, not as a separate afterthought. A lot of EXUI accessibility regressions are caused by shared rendering changes: tab markup, filters, form controls, summary pages, conditional content, modal/dialog behaviour, table structure, focus order, and route-level layout. If those shared surfaces regress, asking every service team to rediscover the same axe failures is wasteful.

Why accessibility belongs here

The same logic applies to accessibility. If a shared XUI component creates an accessibility regression, it should not be discovered separately by every service team. The harness should scan the representative rendered shape, keep the evidence with the functional proof, and make the current baseline honest.

Source of risk

Shared EXUI rendering

  • Common shell, navigation, tabs, filters and layout.
  • CCD-driven event and CYA rendering through the toolkit.
  • Feature flags and service-family config changing what appears on screen.
  • Role and case-type configuration changing available actions.
Harness model

Scan representative shapes

  • Do not run the same accessibility scan thirteen times against the same EXUI component.
  • Pick the slice that exposes the distinct DOM shape: filter, tab, event page, CYA page, table, modal, document viewer.
  • Record which services are covered by that shared shape and which need their own slice.
  • Keep service-specific content issues separate from EXUI-owned component issues.
Evidence

Axe report plus UI proof

  • Use @axe-core/playwright from a shared harness helper so every slice produces the same style of evidence.
  • Attach the accessibility report to the same Playwright/Odhín evidence as the UI proof.
  • Fail the harness when the representative slice has a new axe rule violation or more nodes than the explicit known baseline.
  • Keep screenshots beside the a11y report so reviewers can see the scanned state.
The first accessibility run found real existing issues. That is the point of adding this lane. I would not hide those by disabling axe just to make the demo look clean. The harness records the current known violations as a small explicit baseline, keeps the axe report visible in Odhín, and fails if a future EXUI change introduces anything new or makes the current debt worse.

How this scales without becoming another full SRT burden

1

Identify the shared UI shape

Start from the EXUI change. Ask which reusable screen/component shape changed: task filter, case tab, CYA page, event page, search table, modal, document viewer, or auth page.

2

Map services to that shape

Use EXUI config and CCD definitions to work out which service families exercise the same rendered structure. Group identical shapes; create a new slice only when the DOM or interaction pattern genuinely differs.

3

Scan once per shape

Run the axe scan on the representative harness slice. Do not repeat the same scan per service unless the service has a materially different configuration shape.

4

Attach reviewable evidence

Keep the accessibility report, screenshot, scenario id, service-family grouping, and source anchors together so test leads can inspect why the slice is representative.

5

Escalate real gaps

If a service has custom labels, a unique complex CYA shape, unusual content, or a document-specific accessibility risk, it becomes a new slice or remains service-owned.

Practical rule: accessibility coverage follows the same assurance principle as functional coverage. We cover shared EXUI-owned shapes centrally. We do not claim central coverage for service-specific content, documents, or bespoke journeys unless a matching slice exists.
What this does not prove yet: the first POC does not prove full WCAG compliance for every EXUI page, every service journey, every uploaded document, or every browser/device combination. It proves that accessibility checks can be built into the central harness and attached to the same evidence trail.
Why the baseline approach is honest

The first run found existing accessibility debt. That is useful. The harness should not hide it by switching axe off. It records the known debt explicitly and fails if a future XUI change introduces a new violation or makes the known one worse. That gives test leads a practical gate without pretending the whole product is already clean.

12. The mutation proofs, step by step

In this context, a mutation proof means controlled fault injection. We deliberately remove one expected piece of shared configuration from the observed test contract, then prove the harness catches that exact missing piece with a clear failure message.

1

Control run

Run the focused proof with no mutation. Expected model and observed contract agree. Green.

2

Pick fault class

Use harness:mutation:wa for EXUI-side Work Allocation family drift, or harness:mutation:ccd for CCD-origin search metadata drift.

3

Mutate in memory

The proof removes either PRIVATELAW from the observed WA family list or [CASE_REFERENCE] from PRL search-input metadata. No source file is edited.

4

Assert against model

The assertion compares the observed contract with the central assurance model and source-backed PRL slice.

5

Expected red

The wrapper expects the exact missing-family or missing-search-input failure. If it does not fail, or fails differently, the demo fails.

Exact expected WA failure: api/wa-supported-jurisdiction/get is missing central must-run service families: PRIVATELAW
Exact expected CCD metadata failure: CCD search metadata is missing required EXUI search input fields: [CASE_REFERENCE]
Why this matters: it shows the harness is capable of detecting both a shared EXUI regression in the central WA family contract and a consuming-service CCD metadata drift that would affect EXUI search/workbasket behaviour.
What changes during the demo: only the test process environment changes. The focused proof mutates the observed contract in memory. No EXUI config file, CCD definition file, or harness source file is edited.
Why the red proof is important

A green test run on its own is not enough for this pitch. The red proof shows that the harness is sensitive to the type of shared break we care about. Remove PRL from the Work Allocation contract, or remove the required CCD search input shape, and the harness goes red with a useful message.

13. What we can honestly claim

The boundary of the claim

The strongest version of this proposal is also the safest one: centralise proof where the risk is shared XUI behaviour, and keep service-owned risk with the service. The POC is credible because it is clear about both sides.

Proven by the POC
  • Central EXUI family config can be captured into an executable assurance model.
  • Configured services can be classified so new drift is visible.
  • API checks can protect shared EXUI contracts.
  • Thin UI checks can prove visible rendering once per representative shape.
  • Accessibility checks now run as explicit harness tests on representative UI slices, attach axe reports, and baseline known existing debt instead of hiding it.
  • A deliberate WA regression can be caught and reported with Odhín evidence.
  • A deliberate CCD-origin metadata regression can be caught before PRL search/workbasket surfaces lose required inputs.
Not proven yet
  • Every service business journey is not covered.
  • Every CCD event, page, field, and show condition is not covered; only selected EXUI-facing shapes are represented.
  • Full accessibility compliance across every EXUI route, service journey, document and device is not proven by the first POC.
  • Media Viewer redaction coordinate correctness is not covered.
  • True downstream service defects are not covered by synthetic shims.
  • SRT can only be reduced where service owners accept the representative slice.
